# コンパイラとリンカの挙動（C言語のビルドの魔法）

C言語のコード（`.c` や `.h`）は、そのままではコンピュータが理解できません。
プログラムとして動かすためには `gcc` などのコマンドを使って「翻訳（ビルド）」する必要がありますが、実はこの裏側では **「コンパイラ」** と **「リンカ」** という2つの異なるプログラムによるバケツリレーが行われています。

## 1. プリプロセス（下準備）
ソースコードを翻訳する前の下準備のフェーズです。
* `#include "nfa.h"` などの文を見つけると、そのヘッダーファイルの中身を**そのままソースコードの中にコピペ展開**します。
* `-I` オプションをつけることで、このコピペ元のファイルを探すフォルダを指定することができます。

## 2. コンパイル（個別の翻訳）
次に、**「コンパイラ」**が `.c` ファイルを1つずつ順番に機械語に翻訳し、**オブジェクトファイル (`.o`)** を生成します。

ここで重要なのは、コンパイラは**「他のファイルの実体がどこにあるかは一切知らない」**ということです。
たとえば `test_unit_nfa.c` をコンパイルしているとき、`nfa_compile()` という関数が呼ばれると：
> 「`.h` を見たところ、どこかに `nfa_compile()` って関数があるという**約束**なんだな。今はどこにあるか分からないけど、とりあえずその関数を呼び出すための**空洞（穴）**を開けたまま翻訳しておこう！」

と判断し、未完成のジグソーパズルのような状態で `.o` ファイルを出力します。

## 3. リンク（パズルの合体）
最後に登場するのが **「リンカ (Linker)」** です。
リンカは、コンパイラが作った複数の `.o` ファイルをすべてかき集めて、巨大な1つの実行ファイル（`.out`）を作ります。

```bash
# コマンドの例
gcc tests/test_unit_nfa.c src/cpu/nfa_cpu.c -I./include
```

このコマンドを実行した際、裏側でリンカは以下のような作業をしています：
1. `test_unit_nfa.o` のパズルピースには「`nfa_compile` が入る空洞」があるのを発見。
2. 他のピースを探すと、`nfa_cpu.o` の方に偶然ピッタリ合う「`nfa_compile` の出っ張り（実体）」がある！
3. **ここだ！と言わんばかりにパズルを結合（リンク）させる。**

### もしファイル渡しを忘れると？
もし `gcc` コマンドを書く時に `src/cpu/nfa_cpu.c` を渡し忘れてしまった場合、リンカは空洞を埋める出っ張りを見つけることができません。
この時に発生するのが、プログラマを最も悩ませるエラーの一つ **`undefined reference to ...` （未定義参照エラー：約束した実体が見つからないぞ！）** です。
