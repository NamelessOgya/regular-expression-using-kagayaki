# Thompsonのアルゴリズム：ポインタハックの全貌

## なぜこのテクニックが必要か？
NFA（非決定性有限オートマトン）を構築している最中（例: `a|b` などを処理している時）は、「aを通った後」と「bを通った後」の**両方の出口が、まだどこへ向かえばいいか決まっていない（保留）**という状態になります。

将来、行き先（例えば `c` の部屋）が決まった瞬間に、保留になっていた全員の出口（`s->out`）を一斉に `c` の住所に書き換えてあげたいわけです。通常であれば、この「保留リスト」を管理するために専用の構造体を `malloc` してリストを作る（愚直な実装）必要がありますが、Thompsonのアルゴリズムでは**「全くメモリを新規確保せずに一時リストを作る」**という離れ業をやってのけます。

---

## トリックの核心： union Ptrlist と 3つの主要関数
ポインタハックを実現するために、以下の3つの関数が連携して動作します。

1. **`list1(State **outp)`**
   - **役割:** 新しく作られた未確定の出口（空箱）を「長さ1の未定リスト」として初期化します。
   - **ハック:** `*outp = NULL;` とするだけでなく、その空箱自体の住所を `Ptrlist*` 型として返します。これが鎖の最初の1つになります。
2. **`append(Ptrlist *l1, Ptrlist *l2)`**
   - **役割:** 複数の別々のルート（例: `a`を通るルートと`b`を通るルート）が、将来「同じ一つの目的地」に向かう場合に、それらの未定リストを1つの鎖に束ねます。
   - **ハック:** `l1` の鎖の末尾を探し、そこの空箱の中に「`l2` の住所」を書き込むことで、複数の変数を繋ぐ単方向リストを作ります。
3. **`patch(Ptrlist *l, State *s)`**
   - **役割:** ついに将来の行き先 `s` が確定したとき、束ねておいたリスト全員の出口を一斉に `s` に書き換えます。
   - **ハック:** 鎖を辿りながら、次の鎖の住所を事前に退避した後、空箱の中に「本番の行き先(s)の住所」を上書きします。これによりリスト構造は破壊され、本物のポインタグラフが完成します。

### メモリ上の定義
```c
union Ptrlist {
    Ptrlist *next; // 未定時 (保留リストの鎖として利用)
    State *s;      // 確定時 (本番の行き先ポインタとして利用)
};
```
行き先が未定の時、`State` 構造体の `s->out` は「空箱（ただの8バイト）」です。
Thompsonはこの「本来行き先を入れるための空箱」の中に、**「次に保留になっている別の `s->out` 変数」の住所** を書き込んでしまいます。
これにより、本来の用途とは全く違う「変数の住所をつなぐ鎖（リンクリスト）」が生まれます。

## 具体例： `(a|b)c` （後置表現: `a b | c #`）を構築する過程

### 第1 & 第2段階： `a` と `b` を読んだ直後
`s_a` と `s_b` という2つの「部屋（State）」が malloc されます。この時点では次がどこか分からないため、`out` はそれぞれ鎖の末尾（NULL）として設定されます。

```text
[ s_a の部屋 ]               [ s_b の部屋 ]
  c = 'a'                    c = 'b'
  out = NULL    <--鎖の端     out = NULL    <--鎖の端
```

### 第3段階： `|` (OR記号) を読んだ時！ 🔥マジックの瞬間
ここで分岐用の `s_split` 部屋が作られ、入口が a と b に繋がります。
重要なのはその後です。「どちらを通っても次の行き先は同じだから、未定の矢印を1つに束ねる（`append` する）」という処理が走ります。

`append(&s_a->out, &s_b->out)` では、「aの未定リストの `next` ポインタに、bの未定リストの住所を入れる」という操作が行われます。
メモリ上では、**`s_a->out` の空間に「`s_b->out` 変数の住所（アドレス）」が叩き込まれます。**

```text
[ s_split の部屋 ]
  out  = (s_aへ) ----+
  out1 = (s_bへ) -+  |
                  |  |
      +-----------+  |
      |              |
      v              v
[ s_b の部屋 ]     [ s_a の部屋 ]
  c = 'b'            c = 'a'
  out = NULL         out = [&s_b->out の住所(0x8888など)]
```

`s_a->out` には、確定した Stateポインタではなく、**「別の変数の住所（ポインタへのポインタ）」** が入りました。これで「aの次は決まってないけど、とりあえず次は `s_b->out` も保留になってるからあっち見てね」という**変数の鎖（リスト）**が完成しました。

### 第4段階： 続きの文字 `c` が来て、結合 `#` された時！ 🔥完成の瞬間
ついに「aとbの次の行き先は、cだ！」と確定しました。ここで `patch(束ねたリスト, s_c)` が呼ばれます。

`patch` は鎖を順番に辿って上書きを行います。
1. `s_a->out` を見る。「お、`s_b->out` への鎖だな」と先を退避して覚えた後、**`s_a->out` を `s_c`（本番の住所）で上書き**します。
2. 退避していた先（`s_b->out`）を見る。「お、末尾つまり NULL だな」と覚えた後、**`s_b->out` も `s_c`（本番の住所）で上書き**します。

```text
[ s_split ] ---> [ s_a ] ---> out = [ s_c の住所 ] ----+
             \                                         |
              -> [ s_b ] ---> out = [ s_c の住所 ] -+  |
                                                    |  |
                                    +---------------+  |
                                    v                  v
                                 [ s_c の部屋 ] <------+
                                   c = 'c'
                                   out = NULL (新たな鎖の端に…)
```

見事にリストの鎖構造は上書きされて消し飛び、**「aからもbからもcへ向かう、正しい State ポインタのグラフ」**に生まれ変わりました。

## さらに別例： `a?c` （後置表現: `a ? c #`）を構築する過程

`append` は `|` (選択) だけでなく、`?` (0回または1回) でも活躍します。

### 第1段階： `a` を読み、次に `?` を読んだ時
`a?` とは「aを通る」または「aを避けてスキップする(空ルート)」の分岐です。
分岐用の `s_split` が作られ、出口の1つは `s_a` へ繋がれます。もう1つの出口 `out1` は「空ルート」用として残されます。

ここでも「aを通った後の出口」と「空ルートの出口」の2本は、**どちらも将来の同じ先（この場合は c ）へ向かわなければなりません**。

そこで `append(s_aの未定出口, s_splitの空ルートの未定出口)` が実行されます。メモリ上では以下のように、`s_a->out` という変数の空間に、`s_split->out1` 変数の住所が鎖として書き込まれます。

```text
[ s_split の部屋 ] (入り口)
  out  = (s_aへ) ----+
  out1 = NULL        |  <-- ここが新たな「鎖の端」になる
         ^           |
         |           v
         |      [ s_a の部屋 ]
         |        c = 'a'
         +------- out = [&s_split->out1 の住所]
```

### 第2段階： 続きの文字 `c` が来て確定した時
`c` が来て `patch(束ねたリスト, s_c)` が呼ばれると、`a|b` の時と全く同じように鎖が辿られ、**「s_a->out」と「s_split->out1」の両方の変数が `s_c`（本番の住所）で上書き**されます。

```text
[ s_split の部屋 ] 
  out  = (s_aへ) ----+
  out1 = (s_cへ) --+ | 
                   | | 
                   | v 
                   | [ s_a ] ---> out = (s_cへ) --+
                   |                              |
                   +---------------------------+  |
                                               v  v
                                           [ s_c の部屋 ]
```

このように、演算子が何であれ、「複数の未解決なパス」が発生するたびに `append` で変数の住所を繋ぎ、確定した瞬間に `patch` で一網打尽に本番ポインタで上書きする、という汎用的な仕組みで NFA を構築しています。

## どうして `*` や `+` では `append` を使わないのか？（自己解決の美学）

ここまでの例（`|` や `?`）では、「別々のルートが、将来同じ一つの目的地（例:c）を目指す」ため、出口を `append` で束ねて未来へ託していました。
しかし、**`*`（0回以上）や `+`（1回以上）といった「ループ」を作る場合は、束ねる必要がありません。** その場で `patch` を使って**「過去に繋ぎ直す」**ことで自己解決してしまうからです。

### `*` (0回以上の繰り返し) の場合: `a*`
`a*` は、「aへ行く(ループ)」か「避ける(空ルート)」の分岐です。
分岐用の `s_split` が作られ、出口 `out` が `s_a` へ向けられます。空ルート `out1` は未来(NULL)へ残します。

ここでの最大のマジックは、未来ではなく **「過去への `patch`」** が即座に行われることです。
`patch(s_aの未定出口, s_split)` が実行されるとどうなるでしょうか？
「aを通り終わった後の未定の出口」が、たった今作ったばかりの分岐点 `s_split` に確定接続されます！

```text
      +---------------------------------+ (patchで繋いだ戻るルート)
      |                                 |
      v                                 |
[ s_split の部屋 ] (入り口)           [ s_a の部屋 ]
  out  = (s_aへ) ----------------->  c = 'a'
  out1 = NULL                        out = [ s_split の住所が上書きされた！ ]
         ^
         | これだけが未来(c)へ託すべき「未定の鎖の端」として残る
```

見て分かる通り、`s_a` の未定だった出口は**完全に確定（自己解決）**し、未定リストから消滅しました。
この巨大なループの塊において、未来(c)に向かう出口は `s_split->out1` の**たった1本だけ**になります。出口が1つだけなので、束ねる(`append`)必要は全くないのです。

### `+` (1回以上の繰り返し) の場合: `a+`
`a+` も原理は全く同じです。「最低1回はaを通ってから、もう一度やるか分岐する」という構造になります。
aを通った直後に分岐 `s_split` を置き、`patch` によって「aの出口」を「分岐」へ繋ぎます。
そして、分岐の片方(out)をaの入り口に繋ぎ戻します。

```text
                                +-----------------------------+ (戻るルート)
                                |                             |
(入り口)                         v                             |
[ s_a の部屋 ]               [ s_split の部屋 ]               |
  c = 'a'                     out  = (s_aへ) -----------------+
  out = [s_split の住所]      out1 = NULL (未来へ向かう唯一の空箱)
```

この塊全体の入り口は `s_a` になります。やはり未来に向かう出口は `s_split->out1` の1つだけなので、束ねる必要はありません。

このように、ループの場合は**「未定だった出口を、自分自身（過去）の分岐点に向かってその場で `patch` によって曲げる」**ことで、美しい無限ループを構築しています。

---

## 結論：何が凄いのか？
*   **建設中** は、`s->out` の空間を **「次へ続くリスト（Ptrlist *next）」** として悪用する。
*   **完成時** に、その同じ場所に **「本番の住所（State *s）」** を叩き込む。

1つの変数を、時間差で「全く別の用途」として使い回すことで、メモリ確保（malloc）や解放（free）のオーバーヘッドをゼロに抑えて高速に組み立てを終える。これがC言語のポインタを極めた「ゼロアロケーション（リスト用ポインタに対する）」の究極のコードです。
