# ヘッダーファイル (`.h`) の詳細とルールまとめ

C言語におけるヘッダーファイル (`.h`) は、プログラムを綺麗に分割・管理するために必須の仕組みです。このドキュメントでは、その役割や特有のルールについて解説します。

## 1. ヘッダーファイルとソースファイルの役割分担

C言語では機能ごとにプログラムを分割して書く際、一般的に **ヘッダーファイル (`.h`)** と **ソースファイル (`.c`)** の2つをペアにして作成します。

### ヘッダーファイル (`.h`) : 目次・メニュー表
* **役割**: 「このモジュールにはどんな機能（関数など）があるか」を宣言する場所。
* **特徴**: 実装（中身の処理）は書かず、「関数の名前」「引数の種類」「戻り値の種類」といった**プロトタイプ宣言**だけを記載します。
* **メリット**: 他のファイルがこの機能を呼び出したいときに `#include "..."` することで、事前にその関数の存在を知ることができ、コンパイルエラーを防ぐことができます。

### ソースファイル (`.c`) : 実体・レシピ帳
* **役割**: 実際にコンピュータが実行する処理の詳細を書いた場所。
* **特徴**: ヘッダーファイルで宣言した関数の中身のロジックをここに記述します。

## 2. ヘッダーファイル内での `#define` と「変数の定義」について

`config.h` などのヘッダーファイルの中を見ると `#define MAX_LINE_LENGTH 10000` のような記述があります。一見すると変数のようですが、実は「変数の定義」とは全くの別物です。

* **マクロ定義 (`#define`)**: 変数ではなく、コンパイル前に単なる「文字の置き換え（コピペ）」を行う C 言語の機能です。実行時にメモリ（箱）を消費しないため、配列のサイズなどの「定数」を `.h` ファイルで全ファイル向けに共有する際によく使われます。
* **実際の変数の定義 (`int x = 10;` など)**: 実は、**ヘッダーファイル (`.h`) に変数の定義を書くのは原則 NG（ご法度）** とされています！
  * **理由**: もし複数のソースファイル（`a.c` と `b.c`）が同じ `.h` を `#include` した場合、それぞれのファイルに `int x = 10;` が丸々コピペされてしまいます。最後にそれらを合体（リンク）させる際、コンパイラが「同じ名前の実体（x）が2つあるぞ！」と混乱してエラー（多重定義エラー）を起こしてしまうためです。
  * **正しい共有方法**: 複数のファイルで変数を共有したい場合は、`.h` ファイルには `extern int x;` （xという変数がどこかにあるよ、という**宣言**だけ）を書き、実際のメモリ確保（定義）はいずれか1つの `.c` ファイル内で行います。

## 3. 二重読み込み防止バリア（インクルードガード）

ヘッダーファイル（`.h`）は複数のファイルから `#include` で何度も読み込まれる可能性があるため、中身が二重にコピペされてしまい「再定義エラー」になる事故がよく起きます。これを防ぐおまじないが以下の2つです。

* **`#pragma once`**: ファイルの先頭に1行書くだけで「1回しかコピペしないでね」と指示できるモダンで便利な書き方。
* **`ifndef 〜 define 〜 endif`**: 昔からの伝統的な書き方（`nfa.h` や `utils.h` などに見られる形）。3行必要ですが、どんなに古いコンパイラでも確実に動くという特徴があります。どちらもやっていることは同じです。

## 4. 必須用語：「宣言 (Declaration)」と「定義 (Definition)」の違い

C言語では「これがあるよと知らせるだけ」なのか、「実際に作る」のかで、言葉を厳密に使い分けます。これを理解しておくとエラーメッセージが格段に読みやすくなります。

### ① 宣言 (Declaration) ＝ 「事前申告」
コンパイラに対して「どこかにこういう名前で、こういう形のものがあるからね」と**存在だけを知らせる**ことを指します。メモリは消費しません。`.h` ファイルに書くべきなのはこちらです。
* **プロトタイプ宣言 (Function Prototype)**: 関数の目次。「`int nfa_test(...);` がどこかにあるよ」と伝えること。
* **外部宣言 (External Declaration)**: 変数の目次。「`extern int nstate;` (どこかにこの変数があるよ)」と伝えること。

### ② 定義 (Definition) ＝ 「実体の作成」
実際に具体的な処理を書いたり、メモリ（箱）を確保することを指します。`.c` ファイルに書くべきなのはこちらです。
* **関数の定義**: 「`int nfa_test(...) { ...実際の処理... }`」と中身を書くこと。
* **変数の定義**: 「`int nstate = 0;`」と、実際にコンピュータのメモリ上にその変数の領域を確保すること。

### （例外）マクロと型
* `#define MAX_LINE_LENGTH` による文字置き換え設定 ➔ **マクロ定義 (Macro Definition)**
* `struct` や `typedef` による型の設計図の作成 ➔ **型定義 (Type Definition)** / **型宣言**
（これらは実際にメモリを確保するわけではなくルールや設計図を作るだけなので、例外的に `.h` に「定義」を書いてもOKとされています！）

## 5. インクルードパス（階層を無視できるカラクリ）

`テストファイルなどから #include "nfa.h"` と書いたとき、`../include/nfa.h` のように正確なフォルダ階層を書かなくても呼び出せているのはなぜでしょうか？

これは、C言語のコードを読む時ではなく**「コンパイルする時」にカラクリがあります。**

コンパイル時に渡すコマンド（`gcc` など）には、` -I `（大文字のアイ = Include）という便利なオプション機能があります。
```bash
# 実際のコンパイルコマンドの例
gcc tests/test_unit_nfa.c src/cpu/nfa_cpu.c -I./include
```
この `-I./include` という魔法のオプションをつけることで、コンパイラに対して**「`#include "..."` を見つけたら、デフォルトのフォルダに加えて `./include` フォルダの中も探しに行ってね！」**と指示を出しています。

これにより、プロジェクト内のどの階層のソースコードからでも、短くて綺麗な `#include "nfa.h"` という名前だけで一発で参照できるようになっています。プロの開発現場では標準のテクニックです！
